• Introduction

The main purpose of this SDS document is to provide the outcome of the work of the software architect(s) that is NSU-CLS.This will help to ensure that the final output software product meets the requirements of the end customer, i.e. functions as expected, is reliable, is easy to use, does not demand inordinate efforts to train staff in its use, etc. This document will deliver a detailed description of the software components and sub-systems to be provided as part of the product.

• Design Components

NSU-CLS software has two parts. One is a web application and another is a mobile app. (write about web application). The mobile app has a special feature of voice recognition which enables users to lodge a complaint via speech instead of text. This feature is particularly introduced for the convenience of NSU helpers and staff.

• Class Diagram

Detailed Class Diagram
Detailed Class Description
Class name: Users
Purpose: This class stores all the details as attributes of each user.
Attributes:
i) userUNID - It's a string type data. It stores an unique ID given to each user account by which we can identify it. This is also the primary key of this table.
ii) fullName - It's a string type data. It stores the full name of users
iii) nsuId - It's a string type data. It stores the 10 digits of NSU ID of the users.
iv) email - It's string type data. It stores the email address with which the user registered.
v) password - It's a string type data. It stores the password of the users which he will use to log in.
vi) userType - It's a string type data. It stores the designation of the users (i.e faculty, TA / RA, student, helper etc.)
vii) actorType - It's an enumeration. It states whether the user is a reviewer or non-reviewer. This attribute will be used to determine whether the user will be allowed to review a complaint or not.
viii) accountType - It's an enumeration. It states whether the user logged in via Google Sign In process or the default sign-up and log-in process. This attribute tells us whether a user is verified or not
ix) nsuIdPhoto - It's a string type data. It is used to store the name of the file which is uploaded in backend
x) isVerified - It's a boolean type data. This attribute will determine if the user account is already verified through the email verification process or not. The default value of this attribute is set to false.
xi) active - It's a boolean type data. This attribute will determine if a user account is currently active or it has been deleted by the admin. The active status will be set to false if the admin deletes a user account.
Class name: GoogleVerification
Purpose: This class stores the details of the users who registered using google.
Attributes:
i) googleID - It's a string type data. It stores an unique ID given by google.
ii) isVerified - It's a boolean type data. Its default value is set to true as no email verification process is needed for users logged via google accounts.
Class name: UserVerification
Purpose: This class stores the details of the users who registered using Default Register page.
Attributes:
i) verificationToken - It's a string type data. It is a unique token given to user for verification.
ii) isVerified - It's a boolean type data. Its default value is set to false as all accounts (registered by default method) need to go through the email verification process to be logged in.
iii) expiryDate - It's a Date type data. This verification process has a 24 hour time-limit and will expire after that. This attribute will help to store the expiry time of the given token.
Class name: Complain
Purpose: This class or model helps to store details of each lodged complaint.
Attributes:
i) complainUNID - It's a string type data. It stores an unique ID given to each lodged complaint by which we can identify it particularly. This is the primary key of this table.
ii) complainTitle - It's a string type data. It stores the title or name of the complaint. This attribute is supposed to describe the purpose of the lodge complaint very shortly. This attribute cannot be updated by any user, hence it doesn't need to be a separate model itself.
iii) status - It's an enumeration. It states whether a complaint is open or close. The default value of the status is set to open.
iii) status - It's an enumeration. It states whether a complaint is open or close. The default value of the status is set to open.
iii) edits - It's an integer type data. It stores the number of times a particular complaint is updated. The default value of this attribute is set to 0.
Class name: Comment
Purpose: This class stores all the comments (added by the reviewer of that complaint) of a particular complaint. Since new comments of a particular complaint can be updated, it is kept as a separate class, rather than an attribute.
Attributes:
i) comment - It's a string type data. It stores comment of a complaint added by a reviewer of that complaint.
ii) commentNumber - It's an integer type data. It stores the number of comments of a particular complaint. The default value of this attribute is set to 0. It is also used to get the serial of the comment
Class name: Evidence
Purpose: This class stores details of the evidence which is uploaded for a complain.
Attributes:
i) evidence - It's a string type data. It stores the name of the file which is uploaded as evidence in backend.
ii) editHistory - It's an integer type data. It stores the version number of a particular complaint on which this evidence was uploaded. The default value of this attribute is stored to 0.
Class name: ComplainAgainst
Purpose: This class has an association with the Users class because of which this class has an attribute of Users type that again stores the details of the user against whom the complaint has been lodged.
Attributes:
i) editHistory - It's an integer type data. It stores the version number of a particular complaint on which the user's full name was entered against whom the complaint was lodged. The default value of this attribute is stored to 0.
Class name: ComplainDescription
Purpose: It stores the description of a particular complaint. The description has a limit of 250 words. Since description of a particular complaint can be updated, it is kept as a separate class, rather than an attribute.
Attributes:
i) complainDescription - It's a string type data. It stores the description of a particular complaint.
ii) editHistory - It's an integer type data. It stores the version number of a particular complaint on which this description was updated. The default value of this attribute is stored to 0.
Class name: ComplainReviewer
Purpose: It determines if a particular user(of reviewer type) of a particular complaint is currently the reviewer of it or not. This class is associated with the Users and Complain class because of which this class has an attribute of Users type that again stores the details of the user who is reviewing the complaint; and Complain type that stores the details of the lodged complaint this reviewer is reviewing.
Attributes:
i) currentReviewer - It's an enumeration. It states yes if the reviewer is currently reviewing the complaint and no, if he passed that complaint to another reviewer.

• Database Design

ER Diagram
ER Description
Class name: Users
Associations:
i) This model is associated with the Complain model. It has a one to many relationship as one user can lodge multiple complaints.
ii) This model is associated with the ComplainReviewer model. It has a one to many relationship as one user can review multiple complaints.
Class name: GoogleVerification
Associations:
i) This model is associated with the Users model. It has a one to one relationship with the Users model, as one google verification is for one user only.

UserUNID is the foreign key of GoogleVerification model. It's string type data. The purpose of this is to keep details of the user who registered using google.
Class name: UserVerification
Associations:
i) This model is associated with the Users model. It has a one to one relationship with the Users model, as one user verification is for one user only.

UserUNID is the foreign key of the UserVerification model. It's string type data. The purpose of this is to keep details of the user who registered using register page.
Class name: Complain
Associations:
i) This model is associated with the Users model. It has a one to many relationship with the Users model, as one user can lodge multiple complaints.

ComplainerUNID is the foreign key of Complain model. It's string type data. The purpose of this foreign key is to know the details of the user who has lodged the complaint.

ii) This model is associated with the ComplainReviewer model. It has a one to many relationship as the reviewer of a complaint can be changed and updated.

iii) This model is associated with the ComplainDescription model. It has a one to many relationship as the description of a complaint can be changed and updated.

iv) This model is associated with the ComplainAgainst model. It has a one to many relationship as the complaint can be lodged against multiple users.

v) This model is associated with the Evidence model. It has a one to many relationship as the lodged complaint can have multiple evidence files.

vi) This model is associated with the Comment model. It has a one to many relationship as the reviewer of the complaint can add multiple comments.
Class name: Comment
Associations:
i) This model is associated with the Complain model. It has a one to many relationship as one complaint can have multiple comments.

ComplainUNID is the foreign key of the Comment model. It's string type data. The purpose of this foreign key is to know the details of the complaint of which this comment is added.
Class name: ComplainAgainst
Associations:
i) This model is associated with the Complain model. It has a one to many relationship as one complaint can be lodged against multiple users. This association helps us to keep track of the complain for which the users are added

ComplainUNID is the foreign key of the ComplainAgainst model. It's string type data. The purpose of this foreign key is to know the details of the complaint which has been lodged against this user.

ii) This model is associated with the Users model. It has a one to many relationship as one complaint can be lodged against multiple users.

ComplainAgainstUserUNID is the foreign key of the ComplainAgainst model. It's string type data. The purpose of this foreign key is to know the details of the users against whom the complaint is lodged.
